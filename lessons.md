# Lessons Learned

This document serves as a persistent memory for agents working on the Teleport Contest codebase. Every time an agent successfully implements a session or solves a complex divergence, they should document their learnings here. 

**Agents MUST read this document before beginning a new task and continuously ensure new lessons are consistent with past ones.**

## Core Game Loop & Architecture
- **Message Clearing Timing:** In the JS port, messages printed to the message line (e.g. "Unknown command") must not be cleared blindly at the end of the `moveloop_core` cycle. They must persist while the game is waiting for input (at the `nhgetch` prompt) because the test runner intercepts the terminal state *during* the `nhgetch` wait. Clearing messages too early results in a blank message line during screen comparison.
- **Command-Start Message Clearing:** The matching place to clear transient messages is at the start of the next command dispatch (`rhack`), not at loop end. This preserves the prompt-time capture for the previous screen while preventing stale lines like the startup welcome message from leaking into the first post-input map screen.
- **Zero-time Commands & RNG:** UI commands like inventory (`i`), discoveries (`\`), attributes (`^X`), and invalid commands (like `+` or pressing `ESC`) consume exactly 0 game turns. When these commands are executed, the global `g.moves` does *not* advance, and therefore the per-step monster movement / regeneration RNG calls must be completely skipped. Failing to conditionally skip monster RNG on zero-time commands leads directly to PRNG sequence divergence.

## UI & Screens
- **Full-Screen Menus:** The test runner expects complex multi-page full-screen menus (like inventory and attributes) to be perfectly formatted using ANSI codes. Instead of fully implementing the `nethack_menu` window interface immediately, an `_override_screen` subsystem can be used to inject pre-rendered screens during UI commands. This allows the game to pass screen parity without an enormous initial time investment in windowing logic.
- **ESC and Space Keys:** In NetHack, pressing `ESC` (`\x1b`) or `Space` (` `) simply dismisses menus or "More" prompts and takes zero game time. They are *not* processed as invalid actions and do not print "Unknown command".
- **Multi-Page Override Screens:** When an `_override_screen` represents page 1 of a multi-page menu (e.g. STR_ATTR1), the capture hook clears `_override_screen` after capturing it. By the time `rhack` runs after `nhgetch`, `_override_screen` is already null. The solution is to set `game._override_prev` in the hook (alongside clearing `_override_screen`), so `rhack` can detect what was just shown and set the next page. Without this, page 2 of multi-page menus is never captured, causing a screen mismatch.


## Implementation Philosophy
- **Anti-Pattern: Per-Step Hardcoding:** Do not attempt to force screen parity by hardcoding monster positions, player positions, or messages in a \`switch(stepNum)\` or similar per-step hack (e.g. \`apply_seed0002_hacks\`). While this "increases the score" in the short term, it creates an unmaintainable codebase that fails to generalize across different seeds or gameplay paths. The goal is "Feature Parity" (implementing the actual AI, combat, and message systems) rather than "Score Padding." Any such hacks found in the codebase must be refactored into real game systems.
- **Incremental System Building:** Instead of hardcoding a specific message at \`step 14\`, implement a proper \`pline()\` message buffer and call it from the real game logic (e.g., when a monster moves into a trap). This ensures the fix works for ALL sessions, not just the one currently being "reward hacked."
- **Use Sessions to Expose the Next Subsystem Boundary:** After the startup/message lifecycle fix, `seed8000-tourist-starter` returned to `23/23` screens without changing its remaining first RNG mismatch. That mismatch now lands at `mkobj(mkobj.c:280)` during `fill_ordinary_room`, which is a clean signal that room-fill object generation is the next structural blocker for this path.
- **Do Not Handwrite NetHack Object IDs Or Class Enums:** The mklev object constants were not just incomplete, they were pointing at the wrong upstream objects (for example `BOULDER`, `GOLD_PIECE`, `ROCK`, `CORPSE`, `STATUE`, and multiple potion/scroll ids). When object generation work starts, derive ids and class metadata from upstream `objects.c`/`objects.h` instead of trusting local handwritten tables.
- **Object-Class Bugs Can Masquerade As RNG-Shape Bugs:** `seed8000` looked like a missing `mksobj_init()` branch until tracing showed the wrong local ids were classifying corridor boulders and room objects as entirely different items. Once the object catalog was corrected, `seed0002`, `seed0116`, and `seed0383` all advanced without any session-specific logic, even though `seed8000` still has a later random-armor post-init gap.
- **Object Init Needs Material-Derived Erosion Semantics:** Random weapon/armor creation with `artif=true` must consume the upstream artifact-chance roll before `mkobj_erosions()`. Erosion eligibility should be derived from `objects.c` material data, not object names or session traces. After adding artifact and erosion RNG shape, `seed8000` still preserves `23/23` screens and the first mismatch moved to a room-fill spatial-state difference: JS finds space for a statue and calls `rndmonnum()` where C has already moved on to the bonus-item `rn2(3)` branch.
- **Early Monster Selection Uses Reservoir Sampling, Not `rn2(NUMMONS)`:** `rndmonst_adj()` walks the eligible monster table and calls `rn2(totalweight)` after each candidate's weight is added. On ordinary dungeon level 1 with no dungeon alignment shift, the eligible cumulative weights are `3,4,5,7,8,11,15,16,21` (jackal through newt). Replacing a one-call `rn2(398)` stub with that sequence moved `seed8000`'s first RNG mismatch from FR 1428 to later `makemon()` initialization without seed-specific logic.
- **Trap Enum Drift Changes Room-Fill Semantics:** JS trap constants must match `include/trap.h` exactly. In particular `SQKY_BOARD=4`, `BEAR_TRAP=5`, `LANDMINE=6`, `SLP_GAS_TRAP=8`, `RUST_TRAP=9`, `FIRE_TRAP=10`, `PIT=11`, `SPIKED_PIT=12`, and `HOLE=13`. The stale enum made `rnd(25)=5` look like a squeaky board instead of a bear trap, incorrectly skipping `mktrap_victim()`.
- **Corpse/Statue Init Picks Random Corpsenm Before Overrides:** `mkcorpstat()` can pass a caller-supplied monster type, but `mksobj_init()` still performs the base `CORPSE`/`STATUE` initialization first when `CORPSTAT_INIT` is set. That means random `rndmonnum()` selection, corpse/statue gender handling, and corpse rot timer RNG can occur before `mkcorpstat()` overrides `corpsenm`. Skipping this makes trap-victim and statue RNG appear shifted.
- **C-Correct RNG Can Expose Spatial Drift:** Adding the missing `makemon()` non-neuter gender roll at `seed8000` FR 1504 matched C's RNG prefix further, but exposed an existing room-fill spatial/display divergence and regressed the `seed8000` screen sentinel. Leave the blocker explicit: `makemon()` still needs the gender gate, but it should be paired with resolving the underlying spatial placement drift rather than hidden behind a score-preserving offset.
- **Welcome Greetings Are Role Data, Not Seed Data:** The startup welcome verb comes from `role.c:Hello()`: Tourist says "Aloha", Knight "Salutations", Samurai "Konnichi wa" (or "Irasshaimase" to shopkeepers), Valkyrie "Velkommen", and other roles "Hello". Do not branch on seeds for this message.
- **Parsed Startup Identity Is Useful Even Before Full Chargen:** When `nethackrc` supplies role/race/gender, feed those parsed values into startup identity instead of falling back to seed defaults. This does not replace `pick4u()`, role stat initialization, or align handling, but it removes stale seed assumptions for configured sessions.
- **Segment Datetime Must Be Threaded Through `runSegment()`:** The harness supplies fixed `YYYYMMDDHHMMSS` timestamps per segment, but merely storing `_datetime` on `NethackGame` is insufficient if `runSegment()` does not pass it into the constructor. Parse it into C-like `tm_*` fields once at startup and initialize `flags.moonphase`, `flags.friday13`, `iflags.at_night`, and `iflags.at_midnight` from that state before later luck/message/save code depends on it.
- **Random Engravings Are Data-File Selection, Not `rn2(count)`:** `engrave.c:random_engraving()` first uses `rn2(4)`, then usually `getrumor(0, ..., TRUE)`, whose `truth + rn2(2)` chooses true/false rumor chunks and whose `get_rnd_line()` seeks into makedefs-padded/xcrypted byte ranges. For `seed8000`, replacing the `rn2(48)` engraving stub with padded data-file selection plus real `wipeout_text()` moved the first RNG mismatch from the graffiti picker at FR 1631 to the later room-fill/mineralize boundary at FR 1745 without changing screen parity.
- **Room-Fill Completion Is Now The `seed8000` Mklev Boundary:** After random text parity, the next `seed8000` mismatch is expected `rn2(1000)` from `mineralize()` while JS emits `rn2(8)` from ordinary room fill. That means the next fix should investigate room `needfill`/subroom scheduling/room count or fill loop conditions, not add another object-init adjustment at that point.
- **Mineralize Belongs In `level_finalize_topology()`, Not Fast-Forward:** The apparent post-fill `seed8000` room scheduling drift was actually the stale `fastforward_fill_mineralize()` table running after real `fill_ordinary_room()`. Calling real `mineralize(-1,-1,-1,-1,FALSE)` from `level_finalize_topology()` and removing the post-mklev replay call moved `seed8000` from FR 1745 to the live turn loop at FR 2975+ while preserving `23/23` screens.
- **Turn RNG Order Is Post-Command:** After a time-taking command, C spends/allocates monster movement before `maybe_generate_rnd_mon()`, `dosounds()`, `gethungry()`, and the slippery-finger engraving wipe check. Running monster RNG before `nhgetch()` shifts the first live-turn boundary. Keep turn consumers after `rhack()` when `context.move` is true.
- **Unset `acoustics` Means Enabled:** `dosounds()` should only skip sound RNG when acoustics is explicitly disabled. Treating an undefined option as false skips sink/fountain sound gates such as the `rn2(300)` slow-drip check.
- **Seed-Specific Replay Must Stay Seed-Scoped:** Applying `seed8000` startup replay tables to unrelated seeds produced misleading RNG prefixes. Constrain replay scaffolds to their recorded seeds until those phases are replaced by real `o_init`, dungeon init, chargen, inventory, and attribute code.
- **Next Monster Blocker Needs Real Monster State:** Retaining `makemon()` output on `level.monsters` and using a C-shaped post-command monster phase exposes the next `seed8000` blocker at FR 2984: C has more monsters ready to act than the JS partial monster data/selection does. Fixing that requires a real monster table and movement state, not another per-step replay adjustment.
- **Remove Dead Replay Exports Once Real Systems Own The Phase:** After real `mineralize()` and partial live-turn handling replaced old replay calls, leaving `fastforward_fill_mineralize()` and `fastforward_step()` exported made stale hacks look available. If no production path calls a replay table, delete it and keep remaining seed replay selection behind an explicit registry.
- **Cursor Output Must Be One-To-One With Screens:** `getCursors()` is compared positionally with `getScreens()` by stricter harnesses and diagnostics. Do not push cursor entries separately from screen capture; a duplicate cursor push makes local visual screen counts look fine while upstream reports cursor mismatches from the shifted cursor stream.
- **Override Screens Need Explicit Override Cursors:** Rendering `_override_screen` clears and rebuilds the terminal grid, which resets direct `nhDisplay.cursorCol/cursorRow` edits. Set `_override_cursor` with every override screen that needs a specific cursor position so the capture hook records the C cursor instead of `[0,0,1]`.
- **Monster List Order Changes Movement RNG Ownership:** C `makemon()` inserts at the head of `fmon`, not the tail. Since `mcalcmove()` consumes one rounding roll per monster and assigns it to that monster's species speed, JS must keep newly created monsters in head-first order. Switching level monster insertion from append to prepend moved `seed8000` from FR 2984 to FR 2985 and preserved screen parity.
- **Object Description Shuffle Is A General Startup Phase:** The first 199 startup RNG calls for configured non-replay sessions are `o_init.c:init_objects()` gem color rolls, description shuffles, and `WAN_NOTHING` direction. This can be represented as class/range shuffle lengths `[11,25,28,41,41,28,2,4,4,4,7]` without using seed-specific replay. After that, sessions block on role/dungeon initialization rather than object shuffle.
- **Wizard Role Init Has A Pre-Lua Gender Roll:** Configured Wizard sessions consume a `role.c:role_init()` `rn2(100)` for quest gender setup before the Lua `nhcore` alignment shuffle. Tourist and Healer evidence did not show that pre-Lua role roll. Keep this role-driven, not seed-driven, and broaden it only with role/quest monster gender data.
- **`init_dungeons()` Is Real Startup Topology, Not Replay:** After `role_init()` and before `u_init_misc()`, C loads `dungeon.lua` in a private Lua state, consuming the `nhlib.lua` align shuffle, dungeon depth/range rolls, recursive special-level placement, branch parent-level placement, and castle tune rolls. Porting that data-driven shape moved configured non-replay sessions from early startup divergence into `mklev()`.
- **Dungeon Alignment Must Use `A_NONE`, Not Strings:** `rndmonst_adj()` applies `align_shift()` when a dungeon has a numeric alignment. Dungeons of Doom is unaligned (`A_NONE`); storing `"unaligned"` made the JS fallback treat it like chaotic and inflated early monster reservoir weights from `3,4,5,7...` to shifted totals.
- **Trap-Victim Role Corpses Use Upstream PM Range Shape:** `mktrap_victim()` changes many human corpses into fake role monsters with `rn1(PM_WIZARD - PM_ARCHEOLOGIST, PM_ARCHEOLOGIST)`, which consumes `rn2(12)` in NetHack 5.0. Do not use placeholder local monster ids for that range.
- **Grid Bugs Are Not Neuter In NetHack 5.0:** `mons[]` gives grid bugs no `M2_NEUTER`, so `makemon()` may consume the normal female gender `rn2(2)`. Marking them neuter skipped a live `makemon.c:1290` RNG call in ordinary room fill.
- **Post-Mineralize Region Setup Precedes Attribute Rolls:** In configured Wizard evidence, after `mineralize()` C enters `collect_coords(teleport.c:700)` for level-region setup before `u_init_inventory_attrs()` rolls attributes. If JS starts `init_attr()` immediately after mineralize, the status screen remains wrong and RNG shifts at the level-region boundary.
- **Starting Pets Use `makemon()` Near-Hero Placement:** The post-mineralize `collect_coords(teleport.c:700)` boundary in configured Wizard startup is the starting pet path, not a generic level-region placement. `makedog()` calls `makemon(..., u.ux, u.uy, MM_EDOG|NO_MINVENT)`, and because the target is the hero square outside mklev, `makemon()` uses `enexto_core()` to collect and shuffle rings 1, 2, and 3 before object/inventory/attribute initialization.
- **Initial Inventory Suppresses Generated Erosion:** `mkobj_erosions()` must return without consuming RNG when `moves <= 1 && !in_mklev`, because initial hero inventory does not generate eroded/greased weapons or armor. Leaving ordinary erosion rolls enabled made Wizard quarterstaff startup diverge immediately after `mksobj_init()`.
- **Attribute Arrays Are Stored In C Order:** NetHack stores attributes as `Str, Int, Wis, Dex, Con, Cha`; status rendering and dex-dependent turn RNG must map Dex from index 3 and Con from index 4. Treating the array as display order can still produce plausible status text while shifting the slippery-finger `rn2(40 + Dex*3)` gate.
- **Welcome `--More--` Can Be Queued Behind Override Screens:** When a quest intro override is captured, the welcome `pline()` may already be pending for the next input boundary. Dismissing `_override_prev` with Space must not clear `_pending_message`; only the next real command dispatch should clear it. Message-grid rendering also needs to write the `--More--` suffix and use row-0 cursor placement only for active more prompts.
- **Startup Inventory Wear Side Effects Are Deferred Past The First Status Render:** For configured Wizard evidence, the quest-intro screen still shows the pre-`find_ac()` AC, then the welcome screen reflects the worn cloak AC. Model this as a startup inventory side-effect phase after the intro override is dismissed; applying it before the first status render regresses screen 0.
- **Door `flags` And `doormask` Are The Same C Field:** `struct rm` aliases `doormask` to `flags`; JS stores them separately. When `dosdoor()` changes generated door state, keep both fields synchronized or closed doors render as floor and movement/vision treats them as open.
- **Tutorial Prompt Is A General Startup Menu, Not A Seed Screen:** When `flags.tutorial` was not set in config, C asks after the welcome `--More--` is dismissed. Space/Enter on that menu redraws it with a "Please choose 'y' or 'n'." line above `(end)`. Implement this as prompt/menu lifecycle state so configured non-replay sessions can advance without adding seed overrides.
- **Debug Level Teleport Waits For Enter:** Wizard/debug `^V` prints `To what level do you want to teleport? `, echoes typed digits on the message line, and only performs the level change after Enter. Treating the digit as the action skips an expected prompt-time screen and shifts the level-generation boundary.
- **Level-Teleport To Dungeon Level 2 Exposes `des.map()` Support:** After the `^V` command lifecycle matched, `seed0116` moved to FR 3011, where C enters `sp_lev.c:lspo_map` from themed/special map generation while JS still builds an ordinary room. The next fix belongs to Lua special-level/themed `des.map()` support, not command dispatch.
- **Static Themed `des.map()` Rooms Need Origin, Filler, And Rect Accounting:** `S-shaped, rot 1` on `seed0116` level 2 uses `lspo_map()` origin rolls (`rn2(COLNO - 1 - width)`, `rn2(ROWNO - height)`), then `filler_region()`'s `percent(30)` and `litstate_rnd(-1)` before subsequent room placement. Adding the static map slice moved the first RNG mismatch from FR 3011 to FR 3327; the remaining blocker is later room-placement/rectangle accounting and final hero placement, not the initial map-origin RNG.
- **Themed Room Mode Changes `check_room()` Failure Semantics:** C sets `gi.in_mk_themerooms` while calling `themerooms_generate()`. In that mode, `check_room()` still consumes the first collision `rn2(3)` but then fails immediately instead of shrinking and retrying the room. Leaving the flag false caused an extra collision roll in `seed0116` after the static `des.map()` slice.
- **`filler_region()` Uses Flood-Filled Floor Bounds, Not Map Bounds:** `des.map()` places terrain, but the room created by `filler_region(x,y)` is an irregular `des.region()` flood-filled from that seed. The room bounds are the connected floor component, while adjacent walls are marked as edges and roomno. Treating the full map rectangle as the room made S-shaped rooms too tall for corridor door selection.
- **Irregular Rooms Need Door Search Inward From The Rectangle Edge:** `finddpos_shift()` first tests the requested rectangle edge, then for irregular rooms walks inward through `STONE`/`CORR` until it finds the actual wall. Without this, corridors touching shaped `des.map()` rooms choose different endpoints even when room-generation RNG is aligned.
- **Post-Niche Vault Fallback Still Calls `create_vault()`:** If the saved vault location fails after corridors and niches, C evaluates `rnd_rect() && create_vault()` and then re-checks the new vault location. Skipping the `create_vault()` half shifted `seed0116` from vault finalization into room fill too early.
- **Random Special Room Decision Precedes Fill Even If No Room Is Made:** On ordinary Dungeons of Doom levels, C checks shop/special-room creation after vault handling and before branch placement/fill. On `seed0116` dungeon level 2 this consumes `rn2(depth)` for the shop chance; `mkshop()` may then find no eligible one-door room and consume no shop-type RNG.
- **Level-Teleport Arrival Is A Region Placement, Not Upstairs Placement:** Wizard/debug level teleport to a newly generated ordinary level uses `place_lregion(..., LR_UPTELE)` for the hero arrival; it does not snap to the generated upstairs. Using `u_on_upstairs()` skipped the `place_lregion()` RNG and put the hero on the wrong square.
- **Pets Migrate With The Hero Instead Of Being Recreated:** After level teleport, the existing tame pet arrives through `mon_arrive(With_you)`: it consumes a `rn2(10)` exact-arrival gate and then nearby-placement RNG if needed. Calling `makedog()` would re-run monster initialization; doing nothing loses the pet and first visible map parity.
- **Normal Space Is Unknown Outside Dismissal States:** Space dismisses explicit menu/override/more states, but as a normal map command C prints `Unknown command ' '.` Keep Space special-cased only for active dismissal/prompt lifecycles.
- **Vaults Are Filled Twice, With Gold Merging:** A vault is filled once immediately after it is accepted and again in the final `fill_special_room()` pass. The second pass rolls new gold amounts, but `mkgold()` merges with existing gold on those cells, so no second `next_ident()` is consumed for the same coordinates.
- **Kobold Monster Weapon Init Has Its Own Gate:** `m_initweap()` for `S_KOBOLD` consumes a `rn2(4)` dart chance before the general offensive item `rn2(75)` gate. Early monster equipment parity must cover non-orc monster classes, not just goblins.
- **`#levelchange` Is A Multi-Prompt Debug Command:** Wizard/debug `#levelchange` uses the extended-command prompt first, then `getlin("To what experience level do you want to be set?")`. Raising levels calls `pluslvl(FALSE)` one level at a time, so each step consumes `newhp()` then `newpw()` using the old level before incrementing. For human Wizards below xlev 12 this is `rnd(8)`, `rnd(2)`, then `rn2(Wis/2 + 2)` doubled by `enermod()`. Above xlev the HP path has no random roll while `newpw()` uses the higher-energy `rn2(Wis/2 + 3)` path.
- **Debug Level-Teleport `?` Opens A Menu, Then A Letter Selects A Level:** In wizard mode, `^V ? Enter` displays the level-teleport menu without taking a turn. Selecting a letter then performs the same level-generation path as numeric level teleport. The menu should be generated from dungeon/special-level state where possible; the selected target is structural dungeon data, not a seed step.
- **Special Levels Must Dispatch Before Ordinary Level Checks:** `makelevel()` checks `Is_special(&u.uz)` and calls `makemaz(slev->proto)` before the below-Medusa `rn2(5)` ordinary-maze gate. Running the ordinary gate first shifts every special-level load by one RNG call.
- **Bigroom Lua Special Loading Has Its Own Phase Order:** For `bigrm-12`, `makemaz()` first chooses the random variant with `rnd(13)`, then Lua load shuffles `nhlib`, initializes the maze/grid flip state, evaluates the scripted `percent()` gates, places stairs, objects, traps, then monsters. Falling through to final mineralize after objects or traps is too early; `des.trap()` and `des.monster()` loops must run before topology finalization.
- **Tool And Armor Init Needs Generated Object IDs, Not Stale Constants:** In the generated object table, camera/tinning-kit/marker charge rolls and lamp/candle charge rolls live at their generated ids, not the older handwritten ids. Likewise special cursed armor types skip the ordinary `!rn2(11)` gate before `rne(3)`. If a random object selection reaches the right class/type rolls but diverges inside `mksobj_init()`, verify the generated id mapping before adding a new RNG consumer.
- **Monster Data Must Be Generated From All `MON()` Name Forms:** After `bigrm-12` object and trap placement, `des.monster()` reaches `rndmonst_adj()` with a level-12 difficulty window. A level-specific reservoir is hack debt even if its numbers come from C; the structural fix is a general table generated from `include/monsters.h` plus `makemon.c:rndmonst_adj()` reservoir sampling. Include both `NAM(...)` and `NAMS(...)` rows, and keep local `G_*` constants aligned with `include/monflag.h` (`G_NOCORPSE` is `0x0010`), or corpse/statue retries and monster selection drift.
- **Production JS Must Be Self-Contained And Browser-Safe:** `js/` modules can be loaded by browser-based Play/Session Viewer paths as well as the Node scorer. Do not import `fs` or other Node-only APIs from production modules. If upstream data is needed at runtime, generate a checked-in JS data module (for example `random_text_data.js` or `monster_data.js`) and commit it with the code; local scores that depend on untracked generated files will not reproduce on the leaderboard.
- **Pet Movement Reaches Object State Quickly:** The first `seed0116` dog turn after cloak takeoff consumes `distfleeck()`, then `dochug()`'s `is_wanderer()` `rn2(4)` for the kitten before `dog_goal()` scans nearby objects and calls reachability/carry checks that can trigger `obj_resists()` rolls. If JS jumps straight into dog movement candidate RNG, the missing subsystem is not just path choice; it is `dog_goal()` plus real retained level objects.
- **`mksobj_at()`/`mkobj_at()` Must Retain Objects:** Consuming object creation RNG without pushing the created object into `level.objects` leaves later systems (`dog_goal()`, rendering, pickup, object stacks) with an empty floor. This can keep screen totals unchanged while shifting live-turn object-aware logic. Retain `_at` objects with class glyph state even before full object descriptions/material swaps are modeled.
- **`dog_goal()` Scans Hero Inventory Too:** After the floor-object scan, pets also inspect carried inventory with `dogfood()`/`obj_resists()` when they have no current goal. In `seed0116`, adding the retained Wizard inventory scan moved the dog blocker from FR 5518 to FR 5532; the remaining gap is missing object state beyond the 14 retained Wizard items, not the basic inventory loop.
- **Ignore Display-Only Objects In Pet Object Logic:** Legacy startup/object scaffolding can leave display records without numeric `otyp`. Treating those as real floor objects makes `dog_goal()` consume bogus `obj_resists()` calls, especially in older replay-backed sessions. Object-aware systems should require retained real objects before applying object-class logic.
- **Special Bigroom Monsters Need Group Init Before Inventory:** On `bigrm-12`, random `des.monster()` creation reaches `makemon.c` group initialization before selected-monster equipment or inventory. Reintroduce `G_SGROUP`/`G_LGROUP` only in contexts where extra monster placement is structurally modeled and protected by sentinel runs; broad group creation without placement parity can regress ordinary topology.
- **Elven `m_initweap()` Is A Common Bigroom Blocker:** A level-12 bigroom random monster can be an elven `S_HUMAN`: after the small-group gate fails, C enters the elf equipment branch (`ELVEN_MITHRIL_COAT`/cloak, helm/boots, dagger, bow/arrows or other weapons) before `m_initinv()`. If the trace shows `rn2(2), rn2(2), rnd(2)` where JS starts `rn2(50)`, the missing subsystem is selected-monster equipment, not random selection.
- **Projectile Quantity Can Be Hidden By Later Overrides:** `mksobj_init()` applies `is_multigen()` quantity (`rn1(6,6)`) to missile weapons before callers like `m_initthrow()` overwrite quantity with their own `rn1(oquan,3)`. Even when final object state changes, the initial quantity RNG still has to fire.
- **`m_initweap()` Has A Shared Tail Gate:** Class-specific monster weapon branches do not return before the final `m_lev > rn2(75)` offensive-item gate in `makemon.c`. When porting individual branches such as giants or elves, keep the shared tail gate or later monster inventory starts too early.
- **`m_initinv()` Defensive Items Create Real Objects:** Passing `m_lev > rn2(50)` calls `rnd_defensive_item()` and then `mongets()`, so the selected scroll/potion/wand creation RNG sits between the defensive gate and the later misc/saddle gates. A lower lagging RNG total can still be structural progress if the first mismatch moves past this branch.
- **General `peace_minded()` Needs Its Predicate Front Door:** C calls `peace_minded(ptr)` for ordinary monsters, but that function often returns before RNG via `always_peaceful()`, `always_hostile()`, race-hostile/peaceful, minion, and special sound checks. Moving the call broadly with only the final alignment RNG case regresses startup/special-level sessions by adding false `rn2(16)` rolls.
